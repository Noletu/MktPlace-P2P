## 1. Sistema de Transferência Interna de Criptomoedas

### Contexto
O sistema precisava transferir automaticamente a criptomoeda do vendedor para o comprador quando o pagamento fosse validado.

### Implementação Completa

**Arquivo Principal:** `apps/api/src/services/transaction.service.ts`

**Características:**
- Transação atômica do Prisma (all-or-nothing)
- Verificação de idempotência
- Criação automática de carteira do comprador se não existir
- Registro completo em WalletTransaction (DEDUCT + CREDIT)
- Timeout configurado de 60 segundos
- Logs detalhados no console

**Fluxo de Transferência:**
1. Validação de idempotência (verifica se pedido já foi completado)
2. Busca carteira do vendedor com saldo bloqueado
3. Busca ou cria carteira do comprador
4. DEDUCT do saldo bloqueado do vendedor
5. CREDIT no saldo disponível do comprador
6. Atualização do status da Order para COMPLETED
7. Registra WalletTransactions para rastreabilidade

**Campos Adicionados no Schema:**
```prisma
model Order {
  cryptoTransferred   Boolean   @default(false)
  cryptoTransferredAt DateTime?
  cryptoTransferredTo String?
}
```

### Teste Realizado

**Cenário:** Transação de R$ 500 em BTC com comprador sem carteira

**Resultado:**
- ✅ Sistema criou carteira automaticamente
- ✅ Crypto transferida corretamente
- ✅ Console mostrou: "💸 Crypto transferida: X.XXXXXXXX BTC"

**Feedback do Usuário:**
> "Muito lindo. Fiz a simulação de uma transação de R$ 500 em btc e mesmo com o comprador não tendo a carteira criada o próprio sistema cria. Muito bom"

## 2. Reorganização dos Logs de Auditoria

### Problema Inicial
Logs duplicados de CRYPTO_TRANSFER sem distinção clara entre conclusão da transação e transferência de crypto.

### Solução Implementada

**Arquivo:** `apps/api/src/services/auditLog.service.ts`
- Adicionada constante `ORDER_COMPLETED` (linha 235)

**Arquivo:** `apps/api/src/services/transaction.service.ts`
- Logs reorganizados em 4 tipos distintos (linhas 291-387):
  1. ORDER_COMPLETED para comprador: "Payment validated and approved"
  2. ORDER_COMPLETED para vendedor: "Payment received and confirmed"
  3. CRYPTO_TRANSFER para comprador: "Crypto transferred: X BTC from seller to buyer"
  4. CRYPTO_TRANSFER para vendedor: "Crypto transferred: X BTC from seller to buyer"

**Características:**
- Logs executados fora da transação crítica usando `setImmediate()`
- Descrições unificadas para CRYPTO_TRANSFER
- Metadados incluem: orderId, fromUserId, toUserId, cryptoType, network, amount, direction
- Ordem cronológica: ORDER_COMPLETED → CRYPTO_TRANSFER

### Teste Realizado

**Resultado:**
- ✅ 4 logs criados no timestamp correto
- ✅ Separação clara entre eventos
- ✅ Console: "📝 Audit logs: ORDER_COMPLETED (x2) + CRYPTO_TRANSFER (x2)"

## 3. Correção do Erro 400 no Chat

### Problema Identificado

**Sintoma:** Console mostrava erros 400 repetidos:
```
GET http://localhost:3001/api/v1/chat/order/... 400 (Bad Request)
```

### Análise de Causa Raiz

**Causa #1:** Função `shouldShowChat()` incluía PENDING na lista de status
- Aba Chat era renderizada para pedidos PENDING
- ChatWindow era montado e fazia chamada à API
- Backend rejeitava corretamente com 400

**Causa #2:** Guard em `fetchChatUnreadCount()` não verificava `!order`
- Quando página carrega, `order` é `undefined`
- Guard `order?.status === 'PENDING'` retorna `undefined` (não `true`)
- Código continua e faz chamada à API

### Correções Implementadas

**Arquivo:** `apps/web/app/orders/[orderId]/page.tsx`

**Correção 1 - shouldShowChat() (linhas 703-708):**
```typescript
// ANTES: incluía PENDING e IN_NEGOTIATION
// DEPOIS: chat só a partir de MATCHED
return (
  order.status === 'MATCHED' ||
  order.status === 'PAYMENT_SENT' ||
  order.status === 'VALIDATING' ||
  order.status === 'COMPLETED'
);
```

**Correção 2 - fetchChatUnreadCount() (linha 177):**
```typescript
// ANTES: if (order?.status === 'PENDING')
// DEPOIS: if (!order || order.status === 'PENDING')
```

### Teste Realizado

**Resultado:**
- ✅ Nenhum erro 400 no console
- ✅ Aba Chat não aparece para PENDING
- ✅ Aba Chat aparece após aceitar pedido (MATCHED)

**Feedback do Usuário:**
> "Realmente foi resolvido."

## 4. Análise de Segurança

### Questão do Usuário
> "Qual a criticidade disso? Abre brecha para exploit de alguma maneira?"

### Análise Completa

**Criticidade:** 🟢 BAIXA

**Por que NÃO é uma brecha de segurança:**

1. **Backend está seguro**
   - Validações em `chat.service.ts` bloqueiam acesso não autorizado
   - Verifica se usuário é dono do pedido OU pagador
   - Bloqueia chat para pedidos PENDING

2. **Sem vazamento de dados**
   - Erro retorna apenas: "Chat não disponível para seu próprio pedido"
   - Sem informações sensíveis

3. **Autenticação necessária**
   - Requer Bearer token válido
   - Não explorável publicamente

4. **Sem bypass possível**
   - Mesmo com flood de requests, backend rejeita todas

**Conclusão:**
- ✅ Erro era apenas de UX/Performance
- ✅ Nenhum risco de segurança
- ✅ Backend sempre validou corretamente
