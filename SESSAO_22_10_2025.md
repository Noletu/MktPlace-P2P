## 🐛 Bugs Críticos Corrigidos

### 1. Categorias de Disputa Incompatíveis (Erro 400)

#### **Descrição do Problema**
Usuários não conseguiam criar disputas - sistema retornava erro 400 (Bad Request) independentemente dos dados inseridos.

#### **Investigação**
```
Console Error:
POST http://localhost:3001/api/v1/disputes 400 (Bad Request)
```

Análise revelou incompatibilidade entre enums:
- **Frontend enviava:** `PAYMENT_SENT_NOT_CONFIRMED`, `CRYPTO_NOT_RELEASED`, `FAKE_RECEIPT`, etc. (7 categorias)
- **Backend esperava:** `PAYMENT_NOT_RECEIVED`, `PAYMENT_ISSUE`, `FRAUD`, `OTHER` (4 categorias)

#### **Causa Raiz**
```typescript
// Backend - apps/api/src/controllers/dispute.controller.ts (ANTES)
category: z.enum(['PAYMENT_NOT_RECEIVED', 'PAYMENT_ISSUE', 'FRAUD', 'OTHER'])

// Frontend - apps/web/types/dispute.ts
export enum DisputeCategory {
  PAYMENT_SENT_NOT_CONFIRMED = 'PAYMENT_SENT_NOT_CONFIRMED',
  CRYPTO_NOT_RELEASED = 'CRYPTO_NOT_RELEASED',
  // ... 7 categorias específicas
}
```

#### **Solução Implementada**

**Arquivo 1:** `apps/api/src/controllers/dispute.controller.ts`
```typescript
// DEPOIS - Linha 9-17
category: z.enum([
  'PAYMENT_SENT_NOT_CONFIRMED',
  'CRYPTO_NOT_RELEASED',
  'PAYMENT_NOT_RECEIVED',
  'FAKE_RECEIPT',
  'WRONG_AMOUNT',
  'WRONG_RECIPIENT',
  'OTHER'
]),
```

**Arquivo 2:** `apps/api/src/services/dispute.service.ts`
```typescript
// DEPOIS - Linha 23
category: 'PAYMENT_SENT_NOT_CONFIRMED' | 'CRYPTO_NOT_RELEASED' | 'PAYMENT_NOT_RECEIVED' | 'FAKE_RECEIPT' | 'WRONG_AMOUNT' | 'WRONG_RECIPIENT' | 'OTHER';
```

**Bonus:** Também corrigido validação de attachments:
```typescript
// ANTES
attachments: z.array(z.string().url()).optional()

// DEPOIS
attachments: z.array(z.string()).optional() // Aceita base64
```

#### **Resultado**
✅ Todas as 7 categorias agora funcionam
✅ Upload de anexos em base64 aceito
✅ Usuário validou: "Funcionou direito"

## ✨ Melhorias de UX Implementadas

### 1. Botão "Ver Disputa" em Pedidos Disputados

#### **Descrição**
Quando pedido estava com status `DISPUTED`, não havia forma fácil de acessar a disputa relacionada.

#### **Solução Implementada**

**Arquivo:** `apps/web/app/orders/[orderId]/page.tsx`

**Componente 1 - Estado (linha 67):**
```typescript
const [disputeId, setDisputeId] = useState<string | null>(null);
```

**Componente 2 - Função de Busca (linha 127-151):**
```typescript
const fetchDisputeId = async () => {
  const response = await fetch('http://localhost:3001/api/v1/disputes/my-disputes', {
    headers: { 'Authorization': `Bearer ${token}` },
  });
  const data = await response.json();
  const dispute = data.data.find((d: any) => d.orderId === orderId);
  if (dispute) {
    setDisputeId(dispute.id);
  }
};
```

**Componente 3 - Chamada Automática (linha 101-103):**
```typescript
if (data.data.status === 'DISPUTED') {
  fetchDisputeId();
}
```

**Componente 4 - Atualização canOpenDispute (linha 231):**
```typescript
if (order?.status === 'COMPLETED' || order?.status === 'CANCELLED' || order?.status === 'DISPUTED') {
  return false;
}
```

**Componente 5 - Botão UI (linha 879-887):**
```tsx
{order.status === 'DISPUTED' && disputeId && (
  <button
    onClick={() => router.push(`/disputes/${disputeId}`)}
    className="w-full px-4 py-2 bg-red-600 dark:bg-red-700 hover:bg-red-700 dark:hover:bg-red-800 text-white font-semibold rounded-lg"
  >
    🔍 Ver Disputa
  </button>
)}
```

#### **Fluxo Funcional**
1. Pedido muda para status `DISPUTED`
2. Frontend detecta mudança (polling 5s)
3. `fetchDisputeId()` busca disputa automaticamente
4. Botão "Ver Disputa" aparece na seção Ações
5. Botão "Abrir Disputa" desaparece (não duplicar)
6. Usuário clica → Redireciona para `/disputes/{id}`

#### **Resultado**
✅ Navegação intuitiva
✅ Atualização automática
✅ Visível para ambas as partes
✅ Validado pelo usuário

### 2. Exibição de ID do Pedido

#### **Descrição**
Usuário reportou: "Não consigo ver o ID único para cada pedido"

#### **Investigação**
- ID é gerado: ✅ Sim (CUID via Prisma `@default(cuid())`)
- ID está disponível: ✅ Sim (order.id no frontend)
- ID está visível: ❌ Não (não exibido em nenhum lugar)

#### **Solução Implementada**

**Arquivo:** `apps/web/app/orders/[orderId]/page.tsx` (linha 583-597)

**Componente 1 - ID Visível:**
```tsx
<p className="text-sm text-gray-600 dark:text-gray-400 font-mono">
  Pedido #{order.id.substring(0, 8).toUpperCase()}
</p>
```

**Componente 2 - Botão Copiar:**
```tsx
<button
  onClick={() => {
    navigator.clipboard.writeText(order.id);
    alert('ID completo copiado para área de transferência!');
  }}
  className="text-xs text-blue-600 hover:text-blue-800 dark:text-blue-400 hover:underline"
  title="Copiar ID completo"
>
  📋 Copiar
</button>
```

#### **Design Decisions**

**Por que 8 caracteres?**
- Legibilidade visual
- Suficiente para unicidade (2.8 trilhões de combinações)
- Não polui interface
- Fácil de referenciar verbalmente se necessário

**Por que copiar ID completo?**
- ID curto é só para exibição
- ID completo é necessário para:
  - Buscar em logs
  - Suporte técnico
  - Queries no banco de dados
  - Rastreamento de auditoria

#### **Resultado**
✅ ID visível: `Pedido #CLH7X8Y9`
✅ Botão copiar funcional
✅ Alert de confirmação
✅ Fonte monoespaçada
✅ Dark mode suportado

## 📝 Lições Aprendidas

### 1. Consistência Frontend-Backend
**Problema:** Enums diferentes entre frontend e backend causam erros difíceis de debugar

**Solução:**
- Manter tipos sincronizados
- Criar types compartilhados quando possível
- Validar no frontend ANTES de enviar

### 2. Feedback Visual é Crítico
**Problema:** Validações silenciosas frustram usuários

**Solução:**
- Sempre mostrar limites de caracteres
- Contadores em tempo real
- Botões desabilitados quando inválido
- Mensagens claras

### 3. Navegação Intuitiva
**Problema:** Usuários não sabiam como acessar disputa do pedido

**Solução:**
- Botões contextuais baseados no estado
- Redireccionamentos automáticos
- Ações disponíveis sempre visíveis

### 4. IDs Visíveis
**Problema:** Debugging e suporte difíceis sem IDs

**Solução:**
- Exibir ID curto (legível)
- Botão copiar ID completo (funcional)
- Formato consistente em todo sistema
