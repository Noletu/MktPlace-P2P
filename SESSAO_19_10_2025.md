## Problemas Identificados

### 1. Chat Não Aparecendo Entre Clientes
**Sintoma**:
- Cliente "teste" criou um pedido PIX de R$ 500
- Cliente "teste2" enviou mensagem através do chat
- Cliente "teste" recebeu notificação mas não conseguia ver o botão de chat para responder

**Causa Raiz Identificada**:
- Botão de chat só aparecia para status: `MATCHED`, `PAYMENT_SENT`, `VALIDATING`
- Quando teste2 envia primeira mensagem, o backend muda o pedido para `IN_NEGOTIATION`
- Status `IN_NEGOTIATION` não estava incluído na condição do botão

### 2. Erro 400 ao Buscar Chat via API
**Sintoma**:
```
Failed to load resource: the server responded with a status of 400 (Bad Request)
:3001/api/v1/chat/order/cmgybaeye000k6mxytk4yzqqr
```

**Causa Raiz**:
- Validação no backend (`chat.service.ts:65-67`) impedia owner de acessar chat quando:
  - `isOrderOwner = true`
  - `!isPayer = true` (não é o pagador)
  - `!transaction = true` (não existe transação ainda)
- Em `IN_NEGOTIATION`, ainda não há transaction criada (só é criada quando o pedido é "matched")

### 3. Banco de Dados Desatualizado
**Sintoma**:
```
Error: The column `main.Order.negotiatingUserId` does not exist in the current database
```

**Causa**:
- Múltiplas recriações do banco de dados durante sessão
- Workers falhando porque Prisma Client em memória ainda referenciava schema antigo

### 4. Múltiplas Instâncias de Backend
**Problema**: Havia 2+ processos do backend rodando simultaneamente com conexões ao banco antigo

## Correções Implementadas

### 1. Frontend - Botão de Chat
**Arquivo**: `apps/web/app/orders/[orderId]/page.tsx`
**Linha**: 776

**Antes**:
```tsx
{(order.status === 'MATCHED' || order.status === 'PAYMENT_SENT' || order.status === 'VALIDATING') && (
  <button onClick={() => { setShowChat(true); setChatUnreadCount(0); }}>
    💬 Abrir Chat
  </button>

```

**Depois**:
```tsx
{(order.status === 'IN_NEGOTIATION' || order.status === 'MATCHED' || order.status === 'PAYMENT_SENT' || order.status === 'VALIDATING') && (
  <button onClick={() => { setShowChat(true); setChatUnreadCount(0); }}>
    💬 Abrir Chat
  </button>
)}
```

**Razão**: Incluir status `IN_NEGOTIATION` para permitir que o owner veja o chat quando alguém inicia negociação

### 2. Backend - Validação de Acesso ao Chat
**Arquivo**: `apps/api/src/services/chat.service.ts`
**Linha**: 65

**Antes**:
```typescript
// Impedir owner de criar chat com ele mesmo
if (isOrderOwner && !isPayer && !transaction) {
  throw new Error('Chat não disponível para seu próprio pedido');
}
```

**Depois**:
```typescript
// Impedir owner de criar chat com ele mesmo (EXCETO se já há mensagens/negociação)
if (isOrderOwner && !isPayer && !transaction && order.status !== 'IN_NEGOTIATION') {
  throw new Error('Chat não disponível para seu próprio pedido');
}
```

**Razão**: Se o pedido está em `IN_NEGOTIATION`, significa que alguém já iniciou conversa, então o owner deve ter acesso ao chat mesmo sem transaction

## Lições Aprendidas

### 1. Gestão de Estado do Chat
- Chat precisa estar disponível desde o início da negociação (`IN_NEGOTIATION`)
- Não pode depender da criação de `transaction` (que só acontece após match)

### 2. Validações de Segurança
- Importante permitir que owner acesse chat quando há negociação ativa
- Mas ainda impedir que owner "converse com ele mesmo" em pedidos sem interação

### 3. Banco de Dados SQLite
- Requer atenção ao sincronizar schema durante desenvolvimento
- Prisma Client mantém schema em memória - requer restart quando schema muda
- Melhor abordagem: deletar e recriar completamente quando há problemas

### 4. Debugging de Chat
- Logs do Socket.IO são essenciais para rastrear problemas
- Contador de mensagens não lidas ajuda a identificar se mensagens estão chegando
- Console do browser mostra erros de API claramente
